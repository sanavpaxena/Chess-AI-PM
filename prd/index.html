01 — Product Strategy & Empathy

The "Why" — Starting with the User

AI PMs don't start with the model; they start with the user. Understanding the pain point deeply is what separates a useful feature from a technical demo.

♟

Meet Alex — The Plateau Player

Intermediate Player · 1000–1500 Elo · 2–3 games/day · Blitz & Rapid

Alex has been playing chess for 8 months. After an initial surge from 800 to 1200 Elo, progress has stalled. Alex uses the post-game analysis tool but sees evaluations like "-4.5" without understanding why the move was bad or how to avoid it next time. The engine says "Nf6 was a blunder, best was Bd7" — but doesn't explain what Bd7 achieves.

🎯 1000-1500 Elo 📈 Growth-Oriented 😤 Frustrated by Plateaus 📱 Mobile-First
⚠️ Problem Statement
"Users churn when they hit a plateau because traditional engine evaluations don't provide actionable, human-readable learning. A raw evaluation of -4.5 tells you the position is bad — but not why it's bad, what you should have done instead, or how to recognize similar patterns in future games."

🔄 Data Pipeline Flow

  • ① Fetch — Pull user's latest game from Chess.com public API (PGN format). No API key required. ~200ms latency.
  • ② Analyze — Parse PGN with python-chess. Run Stockfish at depth 16 on each move. Identify the move with the largest centipawn loss (≥200cp = blunder). ~800ms for a 40-move game.
  • ③ Retrieve — Extract board features (material, king safety, pawn structure, themes). Query ChromaDB for the 3 most semantically similar annotated positions. ~50ms.
  • ④ Generate — Construct a RAG prompt with blunder details + retrieved context. Send to Gemini 2.0 Flash via Google AI API (free tier). ~1000ms.
  • ⑤ Validate — Run guardrails: check move legality, detect hallucinated pieces, verify position claims, enforce latency budget (≤2.5s). ~10ms.

⚖️ Key Trade-off: RAG vs. Fine-Tuning

We deliberately chose Retrieval-Augmented Generation (RAG) over fine-tuning a custom model. Here's why:

Dimension RAG Approach ✅ Fine-Tuning Approach
Time to Market Days (plug into existing LLM) Weeks–Months (data prep + training)
Compute Cost $0 (free inference API tier) $100s–$1000s (GPU hours)
Knowledge Updates Add new annotations instantly Requires retraining
Hallucination Risk Grounded in retrieved context Can fabricate chess knowledge
Source Attribution Can cite retrieved sources Black-box generation
Accuracy (Deep) Limited by retrieval quality Can learn chess-specific patterns deeply
Latency ~1.5s (retrieval + generation) ~0.5s (single forward pass)

Decision: RAG is optimal for prototype and MVP. Fine-tuning becomes viable at scale (>100K users) when the investment in training data curation and compute is justified by business metrics.

03 — AI Evaluation & Guardrails

The "Nuance" — Optimizing for UX, Not Just Accuracy

Engineers optimize for accuracy; PMs optimize for the user experience. This section defines how we measure "good" AI and handle edge cases.

🛡 AI Guardrail System

✅

Move Legality Validation

Every move mentioned in the LLM's explanation is validated against the actual position using python-chess. If the LLM suggests an illegal move, it's flagged and the explanation is regenerated or sanitized.

🔍

Piece Existence Check

References to pieces (e.g., "your queen") are cross-validated against the actual board state. If the LLM mentions a piece that doesn't exist in the position, a warning flag is raised.

⏱

Latency Budget Enforcement

Every request is timed against the 2.5s budget. If the LLM takes too long, a pre-computed template response is served instead. Users always get an answer — never a timeout.

🚫

Prompt Injection Protection

User inputs (Chess.com username, PGN) are sanitized before being included in the LLM prompt. The system prompt is isolated and never exposed in the response.

📝

Response Length & Quality

Responses are capped at 350 words. If the LLM generates an empty or nonsensical response (< 20 characters), a fallback template-based explanation is provided. Quality > silence.

🧠 Hallucination Mitigation Strategy

  • 1️⃣ Grounding via RAG — By retrieving actual historical commentary before generation, we significantly reduce the LLM's tendency to fabricate chess knowledge. The prompt explicitly instructs: "Do NOT suggest moves beyond what is provided."
  • 2️⃣ Post-generation validation — Every response passes through the guardrail pipeline (move legality, piece existence, claim verification) before reaching the user.
  • 3️⃣ Graceful degradation — If guardrails detect critical issues, the system falls back to a template-based explanation rather than showing potentially incorrect AI output.
  • 4️⃣ User feedback loop — Thumbs-up/down on each explanation feeds into a monitoring dashboard. Explanations with > 30% negative feedback are flagged for human review.
